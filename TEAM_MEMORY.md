# 团队共享记忆

## 仓库定位
- 这个仓库专门用于维护小红书相关 SOP
- 目标是让不同电脑、不同成员、不同时间都能沿用同一套方法持续更新
- 这里存的是可执行流程，不是随手笔记

## 当前业务背景
- 当前重点平台：小红书
- 当前工作方式：把 SOP、模板、索引、复盘都放进 Git 持续维护
- 当前新增重点：持续迭代“小红书爆款分析”能力
- 使用原则：先读仓库说明，再更新对应 SOP，最后提交并推送

## 协作原则
- 每次开始前先同步最新仓库
- 每次修改尽量只围绕一个明确目标
- 不随意删除旧 SOP，废弃内容移入 `99-归档/`
- 重要改动要写清“为什么改”
- 新增 SOP 时同步更新 `01-总索引/SOP总表.md`

## Codex 工作说明
- Codex 在开始编辑前，先阅读：
  - `TEAM_MEMORY.md`
  - `CURRENT_FOCUS.md`
  - `01-总索引/SOP总表.md`
  - `00-治理规范/命名规范.md`
  - `00-治理规范/更新流程.md`
- Codex 更新 SOP 时，优先复用已有模板和既有结构
- Codex 完成修改后，应同步更新索引或变更记录

## 爆款素材分析规则
- 收到新的爆款素材时，先检查 `02-小红书SOP/08-爆款分析与借鉴/` 里是否已有高度一致或相同的案例。
- 如果发现高度一致或相同案例，先明确告诉用户，并给出已有 Markdown 文件链接，再决定是否新增、合并或跳过。
- 用户发送压缩包素材时，只允许临时解压用于分析；分析完成、Markdown 入库并推送后，删除本次解压产生的本地临时文件。
- 用户保留原始压缩包即可，仓库只沉淀分析 Markdown，不保存解压后的图片、视频、临时文本副本。

## 内容风格偏好
- SOP 要求直接、清楚、可执行
- 优先写“步骤、检查清单、常见问题、优化建议”
- 避免空话、套话、概念化表达
- 尽量让团队成员拿到就能照着做
- 分析类文档要能沉淀出“可复用因子”，不只写结论

## 当前重点分析方向
- 借鉴小红书爆款内容
- 持续整理爆款因子
- 重点关注钩子、开头、标题、封面、结构、互动点
- 分析结果最好能反哺到选题、标题和发布 SOP

## 命名约定
- SOP 文件格式：`SOP-编号_环节_主题.md`
- 新增文件尽量保持中文可读命名
- 不使用“最新版”“最终版”这类无法长期维护的名字

## 维护提醒
- 如果当前重点变化，先更新 `CURRENT_FOCUS.md`
- 如果 SOP 结构变化，补记到 `CHANGELOG.md`
