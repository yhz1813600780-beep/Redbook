# 更新流程

## 日常更新
1. 找到对应 SOP
2. 修改内容
3. 更新文档中的“最近更新”
4. 提交 Git

## 更新前必做
1. 先确认当前仓库已有结构，不要把仓库当成空项目重建。
2. 先搜索相关关键词，检查是否已有同类 SOP、模板、案例或分类归纳。
3. 能补充已有文件时，优先补充已有文件。
4. 只有当现有结构无法承载新内容时，才新增目录或新文档。
5. 新增结构性文件后，要同步更新 README、总索引或 SOP 总表。

## 爆款素材分析更新
1. 收到新素材后，先检查爆款分析目录是否已有高度一致或相同案例。
2. 如果已有相似案例，先告诉用户，并给出已有 Markdown 文件链接。
3. 如果用户发送压缩包，只临时解压用于分析，不把解压文件放入仓库。
4. 分析完成、Markdown 入库并推送后，删除本次解压产生的本地临时文件。
5. 用户保留原始压缩包，仓库只维护分析文档、模板和 SOP。
6. 如果素材能提炼出可复用规律，要同步更新对应的分类爆款因子归纳文件。
7. 如果规律会改变执行方法，再更新相关 SOP，并在 `CHANGELOG.md` 写清楚影响范围。

## 大改动原则
- 写清楚为什么改
- 不直接删除旧方法
- 废弃 SOP 移入 `99-归档/`

## 协作建议
- 先补内容，再优化格式
- 一个提交尽量只做一类修改
- 提交信息直接写清动作和对象
